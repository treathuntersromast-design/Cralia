## 5. エスクロー決済テストシナリオ（将来実装向け）

> Stripe 連携が実装されていない段階では、DB 上の payment レコードを手動で確認・更新することでフローを疑似検証する。

### シナリオ 1: 通常完了フロー

```
TC-U01 (依頼者) → 依頼作成 (draft)
→ TC-U02 (クリエイター) 承諾 (in_progress)
→ Stripe: 依頼者から前払い → payment.status = held
→ TC-U02 納品 (delivered)
→ TC-U01 検収完了 (completed)
→ Stripe: クリエイターへ送金 → payment.status = released
```

**確認SQL（実行後）:**
```sql
SELECT p.status, py.status AS payment_status, py.amount, py.fee
FROM public.projects p
JOIN public.payments py ON py.project_id = p.id
WHERE p.id = 'bbbbbbbb-0002-0002-0002-000000000002';
-- 期待: completed / released
```

### シナリオ 2: 紛争フロー

```
TC-U04 (依頼者) → 紛争申告 (disputed) ← P-003 を使用
→ payment.status = held のまま保留
→ TC-U05 (管理者) が /admin/disputes で確認・仲裁
→ 管理者判断により released または refunded
```

### シナリオ 3: キャンセル・返金フロー

```
TC-U01 → キャンセル申請 (cancel_requested)
→ TC-U02 同意 (cancelled)
→ Stripe: 返金処理 → payment.status = refunded
```
